# [跨架构可移植性问题概述](https://blogs.gentoo.org/mgorny/2024/09/23/overview-of-cross-architecture-portability-problems/)

- 理想情况下，您希望您的程序可以在任何地方运行。不幸的是，即使您使用的是高级“可移植”语言（例如 Python），这也并非易事。在这篇博文中，我想重点介绍我在 Gentoo 期间看到或听到的一些跨架构问题。请注意，我并不是想把这个问题列成一份完整的清单 — 相反，我的目标是让大家读起来有趣。

## 一、哪些因素会破坏 32 位系统上的程序？

### 1、基本整数类型大小

- 如果您问任何人 64 位和 32 位架构之间的主要区别是什么，他们可能会回答是**寄存器大小**。对于许多人来说，寄存器大小意味着基本整数类型的差异，因此当程序仅在 64 位架构上进行测试时（如今这种情况很常见），寄存器大小是 32 位架构上问题的主要来源。实际上，**事情没那么简单**。
- 与普遍预期相反，基本整数类型的差异很小。最重要的是，您的纯文本int在任何地方都是 32 位的。唯一真正不同的类型是long- 在 32 位架构上是 32 位，在 64 位架构上是 64 位。但是，人们在现代程序中**long并不经常使用**，因此不太可能导致问题。
- 也许有些人担心整数大小，因为他们仍然模糊地记得将旧的 32 位软件移植到 64 位架构时出现的问题。正如我之前提到的，int仍然是 32 位 — 但指针变成了 64 位。因此，如果您尝试将指针（或相关数据）转换为int，您将遇到麻烦（因此我们有size_t、ssize_t、ptrdiff_t）。当然，long为 64 位架构制作的同样的东西（即将指针转换为 ）很丑陋，但从技术上讲不会在 32 位架构上造成问题。
- 请注意，我在这里谈论的是 System V ABI。从技术上讲，POSIX 和 C 标准没有指定确切的整数大小，并且允许更多的灵活性（尤其是 C 标准 - 最多可以拥有所有类型，例如，**所有类型都精确为 32 位**）。

### 2、地址空间大小

- 现在，**更可能**出现的问题是地址空间限制。由于**指针**在 32 位架构上是 32 位的，因此程序可以寻址的内存**不超过** 4 GiB（实际上略小于 4 GiB）。这里真正重要的是，这会**限制*分配的*内存**，即使它从未实际使用过。

- 这可能会导致一些奇怪的问题。例如，假设您有一个程序分配了大量内存，但大部分内存都没有使用。如果您在总内存为 2 GiB 的 64 位系统上运行此程序，它会正常工作。但是，如果您在具有大量内存的 32 位用户空间上运行它，它会失败。这是为什么呢？这是因为系统允许程序分配比它所能提供的更多的内存——如果程序实际上试图使用全部内存，**则有OOM 的风险**；但在 32 位架构上，它根本无法将所有这些分配放入 32 位地址中。

- 下面的示例可以简单地证明这一点：

- ```c
  $ cat > mem-demo.c <<EOF
  #include <stdlib.h>
  #include <stdio.h>
  
  int main() {
      void *allocs[100];
      int i, j;
      FILE *urandom = fopen("/dev/urandom", "r");
  
      for (i = 0; i < 100; ++i) {
          allocs[i] = malloc(1024 * 1024 * 1024);
          if (!allocs[i]) {
              printf("malloc for i = %d failed\n", i);
              return 1;
          }
          fread(allocs[i], 1024, 1, urandom);
      }
  
      for (i = 0; i < 100; ++i)
          free(allocs[i]);
      fclose(urandom);
  
      return 0;
  }
  EOF
  $ cc -m64 mem-demo.c -o mem-demo && ./mem-demo
  $ cc -m32 mem-demo.c -o mem-demo && ./mem-demo 
  malloc for i = 3 failed
  
  ```

- 该程序总共分配了 100 GiB 内存，但仅使用每次分配的第一个 KiB。这在 64 位架构上运行良好，但在 32 位架构上由于分配失败而失败。

- 此时，可能值得注意的是，我们正在讨论适用于**单个进程的限制**。32 位内核可以使用超过 4 GiB 的内存，因此多个进程总共可以使用超过 4 GiB 的内存。还有一些糟糕的方法可以让单个进程访问超过 4 GiB 的内存。例如，可以使用memfd_create()（或等效地，文件tmpfs）来创建**超出进程地址空间的内存文件**，或者使用IPC在具有独立地址空间的多个进程之间交换数据（感谢 Arsen Arsenović 和 David Seifert 对此的提示）。

### 3、大文件支持

- 32 位程序面临的另一个问题是文件相关类型传统上是 32 位的。这有两个含义。更明显的一个是off_t，用于表示文件大小和偏移量的类型是一个有符号的 32 位整数，因此您无法stat()打开大于 2 GiB 的文件。不太明显的含义是ino_t，用于表示 inode 号的类型也是 32 位的，因此您无法打开 inode 号为 2^32 及更高的文件。换句话说，如果文件系统足够大，您可能会突然无法打开随机文件，即使它们小于 2 GiB。

- 现在，这个问题可以解决了。现代程序通常定义 D_FILE_OFFSET_BITS=64 并获取 64 位类型。事实上，musl libc 无条件提供 64 位类型，使这个问题成为过去——而且显然 glibc 也计划在未来切换默认设置。

- 这是一个简单的演示：

- ```c
  $ cat > lfs-demo.c <<EOF
  #include <fcntl.h>
  #include <stdio.h>
  #include <unistd.h>
  
  int main() {
      int fd = open("lfs-test", O_RDONLY);
  
      if (fd == -1) {
          perror("open() failed");
          return 1;
      }
  
      close(fd);
      return 0;
  }
  EOF
  $ truncate -s 2G lfs-test
  $ cc -m64 lfs-demo.c -o lfs-demo && ./lfs-demo
  $ cc -m32 lfs-demo.c -o lfs-demo && ./lfs-demo 
  open() failed: Value too large for defined data type
  $ cc -m32 -D_FILE_OFFSET_BITS=64 lfs-demo.c \
      -o lfs-demo && ./lfs-demo
  
  ```

- off_t不幸的是，修复单个包很容易，但全局切换却很难。和的大小会ino_t发生变化，使用这些 API 类型的任何库的 ABI 也会发生变化 — 也就是说，如果您重建库而不重建使用它的程序，它们可能会以意想不到的方式中断。您可以做的是同时切换所有内容，或者慢慢地通过新 API 添加更改类型，保留旧 API 以实现兼容性。后者不太可能发生，因为现在人们对 32 位架构支持的兴趣不大。前者也并非没有问题 — 从技术上讲，您最终可能会引入与使用 32 位类型的预构建软件的不兼容性，并且实际上完全失去了运行某些专有软件的能力。

### 4、time_t 和 y2k38 问题

- 在 C 中，表示时间戳的低级方式是通过自所谓的纪元以来的秒数。这个数字用一个time_t类型表示，正如您可能猜到的那样，在 32 位架构上它是一个有符号的 32 位整数。这意味着它可以保存最多 2 31 – 1 秒的正值，这大致相当于 68 年。由于 POSIX 系统上的纪元定义为 1970 年，这意味着该类型可以表示最多 2038 年的时间戳。

- 这在实践中意味着什么？使用 32 位的程序time_t无法表示截止日期 2038 年之后的日期。如果您尝试执行超出此日期的算术运算（例如“从现在起 20 年后”），则会出现溢出。 stat()对于时间戳超出该点的文件，将会失败（但有趣的是，open()它可以在 glibc 上运行，因此与 LFS 情况并不完全对称）。超过溢出日期，即使尝试获取当前时间，也会出错——如果您的程序没有考虑到time()失败的可能性，它将永远停留在纪元前 1 秒，即 1969-12-31 23:59:59。实际上，它可能最终会随机挂起（等待某个挂钟时间过去），而不是触发事件或用常数播种 PRNG。

- 同样，现代 glibc 版本提供了一个开关。如果您定义_TIME_BITS=64（加上 LFS 标志，作为先决条件），您的程序将获得 64 位time_t。musl libc 的现代版本也默认为 64 位类型（自 1.2.0 起）。不幸的是，切换到 64 位类型会带来与全局切换到 LFS 相同的风险——或者甚至更糟，因为time_t在库 API 中似乎比文件大小相关类型更常见。

- 这些解决方案仅适用于从源代码构建并time_t正确使用的软件。将时间戳转换为int将导致溢出错误。具有 32 位时间戳字段的文件格式本质上是无效的。最重要的是，所有专有软件仍将存在问题，需要认真的解决方法。

- 以下是一些演示问题的示例。请注意，第一个示例假设系统时钟设置在 2038 年之后。

- ```c
  $ cat > time-test.c <<EOF
  #include <stdio.h>
  #include <time.h>
  
  int main() {
      time_t t = time(NULL);
  
      if (t != -1) {
          struct tm *dt = gmtime(&t);
          char out[32];
  
          strftime(out, sizeof(out), "%F %T", dt);
          printf("%s\n", out);
      } else
          perror("time() failed");
  
      return 0;
  }
  EOF
  $ cc -m64 time-test.c -o time-test && ./time-test
  2060-03-04 11:13:02
  $ cc -m32 time-test.c -o time-test && ./time-test
  time() failed: Value too large for defined data type
  $ cc -m32 -D_FILE_OFFSET_BITS=64 -D_TIME_BITS=64 \
      time-test.c -o time-test && ./time-test
  2060-03-04 11:13:32
  $ cat > mtime-test.c <<EOF
  #include <fcntl.h>
  #include <sys/stat.h>
  #include <stdio.h>
  #include <time.h>
  #include <unistd.h>
  
  int main() {
      struct stat st;
      int fd;
  
      if (stat("time-data", &st) == 0) {
          char buf[32];
          struct tm *tm = gmtime(&st.st_mtime);
          strftime(buf, sizeof(buf), "%F %T", tm);
          printf("mtime: %s\n", buf);
      } else
          perror("stat() failed");
  
      fd = open("time-data", O_RDONLY);
      if (fd == -1) {
          perror("open() failed");
          return 1;
      }
      close(fd);
  
      return 0;
  }
  $ touch -t '206001021112' mtime-data
  $ cc -m64 mtime-test.c -o mtime-test && ./mtime-test
  mtime: 2060-01-02 10:12:00
  $ cc -m32 mtime-test.c -o mtime-test && ./mtime-test
  stat() failed: Value too large for defined data type
  $ cc -m32 -D_FILE_OFFSET_BITS=64 -D_TIME_BITS=64 \
      mtime-test.c -o mtime-test && ./mtime-test
  mtime: 2060-01-02 10:12:00
  ```

### 5、这些问题是 C 所特有的吗？

- 值得注意的是，虽然可移植性问题通常是用 C 来讨论的，但并非所有问题都特定于 C，或特定于直接与 C API 交互的程序。

- 例如，地址空间限制会影响所有编程语言，除非它们花费特别的精力去解决这些问题（我不知道有哪一种语言会这样做）。因此，Python 程序将像 C 程序一样受到 4 GiB 地址空间的限制 — 只不过 Python 程序不会明确分配内存，因此限制将取决于使用的内存而不是分配的内存。从不利的一面来看，Python 程序的内存效率可能不如 C 程序。

- 文件和时间类型大小有时也会在内部影响编程语言。现代版本的 Python 启用了大文件支持，因此它们不限于 32 位文件大小和 inode 编号。但是，它们仅限于 32 位时间戳：

- ```python
  >>> import datetime
  >>> datetime.datetime(2060, 1, 1)
  datetime.datetime(2060, 1, 1, 0, 0)
  >>> datetime.datetime(2060, 1, 1).timestamp()
  Traceback (most recent call last):
    File "<stdin>", line 1, in <module>
  OverflowError: timestamp out of range for platform time_t
  ```

## 二、其他一般问题

### 1、字节顺序（字节顺序）

- 如今，主流字节顺序是小端字节序。X86 一直都是小端字节序。ARM 是双端字节序，但默认运行小端字节序（ARM 从来没有太多动力运行大端字节序）。PowerPC 过去默认为大端字节序，但如今 PPC64 系统大多运行小端字节序。

- 并不是说哪种字节顺序在某方面更优越。只是 x86 碰巧随意使用了该字节顺序。鉴于其流行性，许多不可移植的软件只能在小端模式下正确运行。随着时间的推移，人们失去了运行大端系统的动力，这最终导致大端支持总体上更加糟糕。

- 与字节顺序相关的最常见问题发生在实现二进制数据格式时，特别是文件格式和网络协议。缺少字节顺序转换可能会导致程序抛出错误或错误地读取在其他平台上写入的文件、写入错误的文件或无法与其他平台上的对等体正确通信。在极端情况下，缺少一些字节顺序转换的程序可能无法读取之前写入的文件。

- 再次，字节顺序问题不仅限于 C。例如，[Python 中的 struct 模块使用显式的字节顺序、大小和对齐修饰符](https://docs.python.org/3/library/struct.html#struct-alignment)。

- 奇怪的是，字节顺序问题并不局限于低级数据格式。再举一个例子，UTF-16 和 UTF-32 编码也有小端序和大端序变化。当用户没有请求特定的字节顺序时，Python 会使用主机的字节顺序并在字符串中添加BOM，用于在解码时检测正确的字节顺序。

- ```python
  >>> "foo".encode("UTF-16LE") 
  b'f\x00o\x00o\x00' 
  >>> "foo".encode("UTF-16BE") 
  b'\x00f\x00o\x00o' 
  >>> "foo".encode("UTF-16") 
  b'\xff\xfef\x00o\x00o\x00'
  ```

### 2、字符符号

- 这可能是您见过的最令人困惑的可移植性问题之一。粗略地说，问题在于 C 标准没有指定char类型的符号性（与 不同int）。有些平台将其定义为有符号，其他平台将其定义为无符号。事实上，标准更进一步，将其定义为与和char不同的类型，而不是其中任何一个的别名。signed charunsigned char

- 例如，x86 和 SPARC 的 System V ABI 指定char是有符号的，而对于 MIPS 和 PowerPC 则为无符号的。假设任一并在此基础上进行算术运算都可能导致在另一组平台上产生令人惊讶的结果。事实上，我见过的最令人困惑的情况之一是代码仅用于大端平台，因此可以在 PowerPC 上运行，但在 SPARC 上却不行（即使如果在 x86 上使用，它也会失败）。

- 下面是一个受其启发的例子。其基本思想是从char数组中读取一个小端 32 位无符号整数：

- ```c
  $ cat > char-sign.c <<EOF
  #include <inttypes.h>
  #include <stdint.h>
  #include <stdio.h>
  
  int main() {
          char buf[] = {0x00, 0x40, 0x80, 0xa0};
          char *p = buf;
          uint32_t val = 0;
  
          val |= (*p++);
          val |= (*p++) << 8;
          val |= (*p++) << 16;
          val |= (*p++) << 24;
  
          printf("%08" PRIx32 "\n", val);
  }
  EOF
  $ cc -funsigned-char char-sign.c -o char-sign
  $ ./char-sign
  a0804000
  $ cc -fsigned-char char-sign.c -o char-sign
  $ ./char-sign
  ff804000
  ```

- 请注意，为了演示，示例使用-fsigned-char并-funsigned-char切换以覆盖默认平台符号。在实际代码中，您应明确使用unsigned char。

### 3、严格对齐

- 我觉得对齐并不是一个众所周知的问题，所以也许我应该先解释一下。长话短说，对齐就是确保特定类型放置在适当的内存边界上。例如，在大多数平台上，32 位类型应该在 32 位（= 4 字节）边界处对齐。换句话说，您希望类型的内存地址是 4 字节的倍数——无论它是在堆栈还是堆上，直接使用，在数组、结构或结构数组中使用。

- 也许最简单的解释方法是展示编译器如何实现结构对齐。请考虑以下类型：

- ```c
  struct {
      int16_t a;
      int32_t b;
      int16_t c;
  }
  ```

- 如您所见，它包含两个 2 字节类型和一个 4 字节类型——总共 8 个字节，对吗？这没什么不对的，至少在需要 32 位对齐的平台上是这样int32_t。为了保证b在整个结构正确对齐时也能正确对齐，编译器需要将其移动到 4 的倍数的偏移量。此外，为了保证如果结构在数组中使用，每个实例都正确对齐，它还需要将其大小增加到 4 的倍数。

- 实际上，最终的结构类似于以下内容：

- ```c
  struct {
      int16_t a;
      int16_t _pad1;
      int32_t b;
      int16_t c;
      int16_t _pad2;
  }
  ```

- 事实上，你会发现有些库实际上定义了具有显式填充的结构。因此，b在偏移量 4 处，填充为 2 + 2 字节，总大小为 12 字节。

- 现在，如果不满足对齐要求会发生什么？在大多数平台上，未对齐的类型仍会起作用，通常会导致性能下降。但是，在某些平台（如 SPARC）上，它们实际上会导致程序以 终止SIGBUS。请考虑以下示例：

- ```c
  $ cat > align-test.c <<EOF
  #include <inttypes.h>
  #include <stdint.h>
  #include <stdio.h>
  
  int main() {
  	uint8_t buf[6] = {0, 0, 0, 4, 0, 0};
  	int32_t *number = (int32_t *) &buf[2];
  	printf("%" PRIi32 "\n", *number);
  	return 0;
  }
  EOF
  $ cc align-test.c -o align-test
  $ ./align-test
  1024
  ```

- 该代码旨在模仿一种从文件读取数据的廉价方式，然后在偏移量 2 处获取 32 位整数。但是在 SPARC 上，此代码无法按预期工作：

- ```c
  $ ./align-test
  Bus error (core dumped)
  ```

- 您可能已经猜到了，有相当多的程序会遇到这样的问题，仅仅是因为它们不会在 x86 上崩溃，而且很容易消除正常的编译器警告（例如通过类型双关，如示例中所用）。但是，如前所述，此代码不仅会导致 SPARC 崩溃 — 还可能导致其他地方的性能下降。

### 4、堆栈大小

- 低级 C 程序员通常会了解到，程序可用的内存主要有两种：堆和堆栈。堆是进行显式分配的主要内存区域。堆栈是提供给程序立即使用的相对较小的内存区域。
- 主要区别在于堆的使用是受控的——编写良好的程序会分配所需的内存，并且不会访问超出范围的区域。另一方面，堆栈的使用是“不受控制的”——程序通常不会检查堆栈边界。正如您可能猜到的那样，这意味着如果程序使用过多，它将超出可用堆栈——即发生堆栈溢出，这通常表现为“奇怪的”分段错误。
- 那么你实际上如何使用大量堆栈内存？在 C 语言中，局部函数变量保存在堆栈中 — 因此你使用的变量越多，你填充的堆栈就越多。此外，一些 ABI 使用堆栈来传递函数参数和返回值 — 例如 x86（但不是较新的 amd64 或 x32 ABI）。但最重要的是，堆栈框架用于记录函数调用历史记录 — 这意味着调用越深，堆栈使用量就越大。
- 这正是程序员被警告不要使用递归算法的原因——尤其是如果在构建时没有对深度递归进行保护，它们会很容易导致堆栈溢出。最后一个问题并不局限于 C——Python 中的递归函数调用也会导致 C 中的递归函数调用。Python 带有默认的递归限制，以防止这种情况发生。然而，正如我们最近发现的，这个限制需要在不同的架构和编译器配置之间进行调整，因为它们的堆栈[帧大小可能有很大差异](https://github.com/python/cpython/pull/124264#issuecomment-2363825291)：从 x86 或 ARM 等常见架构上的 8-16 字节基线，到 PPC64 上的 112-128 字节，再到 s390x 和 SPARC64 上的 160-176 字节。
- 除此之外，[默认线程堆栈大小在标准 C 库中有所不同](https://wiki.musl-libc.org/functional-differences-from-glibc.html#Thread-stack-size)。在 glibc 上，它通常在 2 MiB 到 10 MiB 之间，而在 musl 上则是 128 KiB。因此，在某些情况下，您可能实际上需要明确请求更大的堆栈。

## 三、浮点类型的奇妙世界

### 1、x87 数学

- x86平台支持两种浮点运算模式：

  - 采用 80 位精度寄存器的传统 387 浮点算法（-mfpmath=387）。
  - 更现代的 SSE 算法支持所有 32 位、64 位和 80 位精度类型（-mfpmath=sse）。

- 前者是使用 System V ABI 的 32 位 x86 平台上的默认设置，而后者则是其他所有平台上的默认设置。这为什么重要？因为前者可能意味着在将结果转换回原始类型之前使用扩展的 80 位精度执行一些计算，这实际上意味着舍入误差小于直接在原始类型上执行相同计算。

- 请考虑以下示例：

- ```c
  $ cat > float-demo.c <<EOF
  #include <stdio.h>
  
  __attribute__((noipa))
  double fms(double a, double b, double c) {
  	return a * b - c;
  }
  
  int main() {
  	printf("%+.40f\n", fms(1./3, 1./3, 1./9));
  	return 0;
  }
  EOF
  $ cc -mfpmath=sse float-demo.c -o float-demo
  $ ./float-demo
  +0.0000000000000000000000000000000000000000
  $ cc -mfpmath=387 float-demo.c -o float-demo
  $ ./float-demo
  -0.0000000000000000061663998560113064684174
  ```

- 这里发生了什么？程序正在计算1/3 * 1/3 - 1/9，我们知道它应该是零。但使用 x87 FPU 指令时则不是。为什么？

- 通常，此计算分两步完成。首先，1/3 * 1/3进行乘法。然后，1/9从结果中减去。在 SSE 模式下，这两个步骤都直接在double类型上完成。但是，在 x87 模式下，双精度数首先转换为 80 位浮点数，然后对这些浮点数进行两次计算，然后将结果转换回双精度数。我们可以看到，查看相应的汇编片段：

- ```c
  $ cc -mfpmath=sse float-demo.c -S -o -
  […]
  	movsd	-8(%rbp), %xmm0
  	mulsd	-16(%rbp), %xmm0
  	subsd	-24(%rbp), %xmm0
  […]
  $ cc -mfpmath=387 float-demo.c -S -o -
  […]
  	fldl	-8(%rbp)
  	fmull	-16(%rbp)
  	fsubl	-24(%rbp)
  	fstpl	-32(%rbp)
  […]
  ```

- 现在，⅓ 和 ⅑ 都无法用二进制精确表示。因此，1./3实际上是 ⅓ + 一些误差，而1./9⅑ + 另一个误差。1./3 * 1./3舍入后，得到的值与 相同1./9— 因此用一个减去另一个得到零。但是，当使用中间类型的更高精度进行计算时， 的平方误差1./3 * 1./3会以更高的精度舍入 — 因此与 中的平方误差不同1./9。因此，与直觉相反，此处的更高精度会放大舍入误差并产生“不正确”的结果！

- 当然，这不是什么大问题——我们谈论的是小数点后 17 位，面向用户的程序可能会将其四舍五入为 0。但是，这可能会导致编写的需要精确值的程序出现问题——例如在测试套件中。

- Gentoo 已将 amd64 多库配置文件切换为强制-mfpmath=sse用于 32 位版本，并且还计划切换 x86 配置文件。虽然这不能解决根本问题，但它会在不同的架构中产生更一致的结果，从而降低我们的用户遇到这些错误的风险。然而，这有一个令人惊讶的缺点：一些软件包实际上适应了在 32 位 x86 上期望不同的结果，现在当在那里使用 SSE 算法时会失败。

### 2、不需要两个架构就可以解决舍入问题

- 实际上，您不必在两种不同的架构上运行程序即可看到舍入问题 - 不同的优化级别，特别是 CPU 指令集也会导致不同的舍入误差。让我们尝试使用和不使用 FMA 指令编译前面的示例：

- ```c
  $ cc -mno-fma -O2 float-demo.c -o float-demo
  $ ./float-demo
  +0.0000000000000000000000000000000000000000
  $ cc -mfma -O2 float-demo.c -o float-demo
  $ ./float-demo
  -0.0000000000000000061679056923619804377437
  ```

- 第一次调用与之前大致相同。第二次调用允许使用 FMA 指令集，该指令集可一步执行乘法和减法：

- ```c
  $ cc -mfma -O2 float-demo.c -S -o -
  […]
  	vfmsub132sd	%xmm1, %xmm2, %xmm0
  […]
  ```

- 再次，这意味着中间值的舍入不会向下舍入为double—，因此不会产生与 相同的误差1./9。

- 底线是：永远不要完全匹配浮点计算结果，要允许一些错误。即使某些方法对你有用，它也可能不仅对不同的架构失败，甚至对不同的优化标志也失败。与直觉相反，更精确的结果可能会放大错误并产生直观的“错误”值。

### 3、长双精度类型

- 您现在可能已经猜到了，C 标准并没有准确定义float、double和long double类型。幸运的是，前两种类型似乎分别统一实现为单精度（32 位）和双精度（64 位）IEEE 754 浮点数。但是，就第三种类型而言，我们可能会发现它是以下任何一种：

  - double与 32 位 ARM 等架构上的类型相同，
  - 80 位 x87 扩展精度类型 — 在 amd64 和 x86 上，
  - 一种实现双精度运算的类型 - 即将数字表示为两个double值的总和，给出大约 106 位的精度，例如在 PowerPC 上，
  - 四倍精度（128 位） IEEE 754 类型 — 例如在 SPARC 上。

- 再次强调，这主要是精度问题，因此它只会破坏假设类型具有特定精度的测试套件。为了演示精度的差异，我们可以使用以下示例程序：

- ```c
  #include <stdio.h>
  
  int main() {
  	printf("%0.40Lf\n", 1.L/3);
  	return 0;
  }
  ```

- 在不同的架构上运行它，我们将看到：

- ```c
  arm64：0.3333333333333333333333333333333333172839 
  ppc64：0.3333333333333333333333333333333292246828 
  amd64：0.33333333333333333333423683514373792036167 
  arm32：0.3333333333333333148296162562473909929395
  ```

## 四、概括

- 可移植性并非小事，这一点很明显。也许更令人惊讶的是，可移植性问题并不局限于 C 和类似的低级语言——我已经展示了多个例子来说明它们如何渗入 Python。
- 如今最常见的可移植性问题可能来自 32 位架构。如今许多项目仅在 64 位系统上进行测试，因此在 32 位平台上面临回归问题。也许令人惊讶的是，大多数问题并非源于 C 中的错误类型使用，而是源于平台限制 — 可用地址空间、缺乏对大文件或大文件的支持time_t。所有这些限制也适用于基于 C 运行时构建的非 C 程序，有时需要进行非平凡的修复。值得注意的是，切换到 64 位time_t将是一个重大的突破性变化（我将在另一篇文章中介绍）。
- 其他问题可能更加模糊，并且特定于各个架构。在 PPC64 或 SPARC 上，我们遇到了与大端字节顺序相关的问题。在 MIPS 和 PowerPC 上，我们可能会对char无符号感到惊讶。在 SPARC 上，如果我们没有正确对齐类型，我们就会遇到崩溃。同样，在 PPC64 和 SPARC 上，我们也更有可能遇到堆栈溢出。而在 i386 上，我们可能会发现由于浮点计算的精度不同而导致的问题。
- 这些只是一些例子，绝对不会排除所有可能的问题。此外，有时您可能会发现两个不同问题的组合，这会进一步加深您的困惑 — 就像仅在使用signed 的大端系统上损坏的软件包一样char。
- 另一方面，所有这些差异提供了一个有趣的机会：通过在一系列架构上测试软件包并了解它们的特性，您可以猜测它可能存在什么问题。比如，如果它在 PPC64 上失败但在 PPC64LE 上通过，您可能会猜测这是一个字节顺序问题 - 然后事实证明它实际上是一个堆栈溢出，因为大端 PPC64 恰好默认使用使用稍大堆栈框架的 ELFv1 ABI。但是，嘿，通常它确实有帮助。
- 可移植性很重要。有问题的架构可能只占用户群的一小部分——事实上，有时我确实怀疑我们正在修复的一些程序是否真的会被这些架构的任何实际用户使用，或者我们只是在滥用很久以前添加的关键字。你甚至可能会争辩说，如果人们丢弃这些机器而不是让它们继续消耗能源，对环境更好。然而，**可移植性造就了好的代码**。今天似乎困扰极少数人的事情，明天可能会防止所有用户发生重大安全事故。
