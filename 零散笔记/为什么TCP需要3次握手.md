# [为什么TCP需要3次握手](https://www.pixelstech.net/article/1727412048-Why-TCP-needs-3-handshakes)

# 预备知识

- 首先我们来了解一下TCP的控制位和状态机，它们是理解TCP三次握手的基础。

## 一、TCP 数据包控制位

- TCP **报文**头中的控制位用于控制TCP连接的状态，可以指示连接建立、终止、重置等各种控制信息。
- 有六个公共控制位：
  - **SYN**（同步序列号）：请求建立连接（三次握手的一部分）。它在连接建立期间的初始数据包中设置，表示发送方希望建立连接并同步序列号。由于 TCP 是双向的，因此双方在建立连接时都需要发送**SYN 。虽然****SYN**数据包不能携带数据，但它们确实会消耗序列号。
  - **ACK**（确认字段有效）：确认收到的数据。设置**ACK**标志后，接收方会在确认字段中填写下一个预期序列号。不携带数据的**ACK**数据包不会占用序列号。
  - **FIN**（发送方不再发送数据）：请求终止连接（四次挥手告别的一部分）。在发送完所有数据后，它会被设置在数据包中，通知接收方发送方已发送完所有数据。由于 TCP 是双向的，因此双方在关闭连接时都需要发送**FIN 。虽然****FIN**数据包不携带数据，但它们会消耗一个序列号。
  - **RST**（Reset the Connection）：重置连接，用于异常或错误终止连接。当接收方收到**RST**标志时，它会立即终止连接而不进行任何数据确认。例如，之前文章中提到的**TIME_WAIT**问题。注意：在生产环境中出现**RST数据包通常表示存在潜在问题。**
  - **PSH**（推送功能）：指示接收方应立即将收到的数据传送到应用层，建议快速处理数据而不是等待更多后续数据。
  - **URG**（紧急指针字段重要）：表示数据具有高优先级，应尽快由接收方处理。
  - **ECE**（ECN-Echo）：表示双方在三方握手过程中是否协商支持显式拥塞通知。

## 二、序列号 (Seq)

- 由于 TCP 是双向的，单个连接的两侧都可以互相发送数据，因此两侧都必须维护自己的**Seq**字段。
- ![](..\images\seq.jpg)

- Seq是动态随机生成的**，**有助于防止伪造的数据包重置连接（RST 攻击）。

- TCP提供有序传输，因此每个数据段必须包含一个**Seq**号字段：

  - 当接收方收到无序的数据包时，它可以根据 Seq 对其进行重新排序。
  - 当接收方收到重复的数据包时，它可以根据 Seq 对其进行重复数据删除。

- 如图，序列号的递增方法（**重要**）：

  - 对于段 1，如果起始**Seq**号为 1 且长度为 1448 字节，则段 2 的**Seq**号为 1 + 1448 = 1449。
  - 如果段2的长度也是1448，那么段3的**Seq**号就是1449+1448=2897。

  换句话说，**Seq**号的大小是通过将**Seq**号与前一个段的长度相加得出的。
